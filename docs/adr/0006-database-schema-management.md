# 6. Database Schema Management and Migrations

Date: 2026-07-21

## Status

Accepted

## Context

PitWall AI relies on PostgreSQL for relational metadata (Seasons, Circuits, Drivers) and TimescaleDB (an extension of PostgreSQL) for high-frequency time-series telemetry data (Laps, Sectors, Telemetry, Weather).

As the application evolves, the database schema will change. We need a robust, automated way to manage schema migrations across development, staging, and production environments, while fully supporting TimescaleDB specific features like hypertables, compression policies, and continuous aggregates.

## Decision

We will use **SQLAlchemy 2.0** as our ORM and **Alembic** as our migration management tool, wrapped inside a dedicated workspace library `libs/models`.

### Architecture

1. **Shared Library (`libs/models`)**: All database models, both relational and time-series, will be defined centrally in `libs/models`. This guarantees that the Storage Sink, Analytics engine, and API services all interact with the same exact definitions.
2. **Alembic Migrations**: The `alembic` environment will reside within `libs/models`. Migrations will be tracked in source control.
3. **TimescaleDB Integration**:
   - Since Alembic's autogenerate feature does not natively understand TimescaleDB hypertables, the creation of hypertables (`SELECT create_hypertable(...)`) and compression policies will be manually added to the migration scripts generated by Alembic.
   - We will structure the SQLAlchemy models to clearly differentiate between relational tables (standard) and hypertable-backed time-series models (no standard integer primary key, indexed primarily by timestamp).

### Migration Execution

Migrations are applied differently depending on the environment:
- **Development**: Developers will run `alembic upgrade head` locally via `uv run` or a `Makefile` target.
- **Production/Docker**: A dedicated, ephemeral `migration` container will be defined in `docker-compose.yml`. This container will run the migration scripts against the database during startup/deployment, before the dependent services (e.g., Storage Sink) are allowed to start.

## Consequences

- **Positive:** Centralizing models in `libs/models` prevents schema drift and code duplication across microservices.
- **Positive:** Alembic provides a proven, robust way to manage PostgreSQL schema evolution.
- **Negative:** TimescaleDB specifics (hypertables, retention policies) require manual intervention in Alembic scripts, as autogenerate will not catch them. Developers must be trained to review and modify generated migrations to add TimescaleDB functions.
